================================================================================
7  DESENVOLUPAMENT
================================================================================

El projecte CityFix s'ha desenvolupat seguint la metodologia Scrum adaptada
al context individual d'un TFG, organitzant el treball en Sprints de dues
setmanes cadascun. Cada Sprint ha produït un increment funcional verificable,
assegurant que les funcionalitats més crítiques reposessin sobre una base
sòlida abans d'afegir-hi noves capes de complexitat.

El sistema està compost principalment per tres grans components: un backend
(servidor Express amb TypeScript), un frontend web d'administració (React
amb Tailwind CSS) i una base de dades PostgreSQL allotjada a Supabase amb
l'extensió PostGIS per a les capacitats geoespacials. El frontend s'encarrega
de presentar les dades i permetre la interacció de l'administrador, mentre
que el backend processa les peticions, aplica la lògica de negoci i gestiona
la persistència de dades.

A continuació es detalla el treball realitzat en cada etapa del desenvolupament,
seguint l'ordre cronològic dels Sprints executats fins al moment.


--------------------------------------------------------------------------------
7.1  Sprint 2: Infraestructura Backend i Lògica de Control
--------------------------------------------------------------------------------

El segon Sprint ha estat fonamental per establir els ciments tècnics del
projecte. L'objectiu principal era disposar d'una API REST funcional capaç
de gestionar l'autenticació d'usuaris, el control d'accés basat en rols i
el cicle de vida complet de les incidències mitjançant una màquina d'estats
finits.

7.1.1  Arquitectura del servidor

L'arquitectura del backend segueix el patró Routes - Middleware - Controller -
Service - Database, on cada capa té una responsabilitat clarament definida.
Aquesta separació de responsabilitats facilita la mantenibilitat i l'escalabilitat
del sistema, ja que permet modificar la lògica de negoci sense afectar les
capes de presentació o d'accés a dades.

El punt d'entrada de l'aplicació (index.ts) configura la instància d'Express,
registra els middlewares globals, munta els grups de rutes sota el prefix /api
i estableix la connexió amb la base de dades. S'ha definit un endpoint de
verificació de salut (GET /api/health) que permet comprovar ràpidament que
el servidor està operatiu.

Per a la gestió de la configuració, s'ha creat un mòdul dedicat que carrega
les variables d'entorn amb dotenv i les exposa com a objecte tipat. La
validació de la cadena de connexió a la base de dades es realitza al moment
de la importació, aturant el procés immediatament si no està definida. Aquesta
decisió de disseny evita que el servidor arrenqui en un estat inconsistent.

La connexió a PostgreSQL s'ha implementat mitjançant Prisma v7 amb el driver
adapter PrismaPg. A diferència de versions anteriors, Prisma v7 requereix un
adapter que encapsula la connexió amb el driver natiu pg, proporcionant millor
rendiment i compatibilitat amb les funcionalitats avançades de la base de dades.

7.1.2  Sistema d'autenticació i control d'accés

La seguretat de l'API s'implementa a través de dos middlewares encadenats que
operen de manera complementària:

El primer middleware (authenticate) s'encarrega de la verificació dels tokens
JWT. En rebre una petició, extreu el token del header Authorization, verifica
la seva signatura criptogràfica mitjançant jwt.verify() i, si és vàlid,
injecta el payload descodificat (identificador d'usuari i rol) a l'objecte
de la petició. Si el token no existeix, ha estat manipulat o ha expirat, la
petició es rebutja amb un codi 401.

El segon middleware (authorize) implementa el control d'accés basat en rols
(RBAC). Es tracta d'una funció d'ordre superior que rep una llista de rols
permesos i retorna un middleware que verifica si l'usuari autenticat posseeix
un dels rols requerits. En cas contrari, retorna un codi 403. Aquest disseny
permet protegir rutes de manera declarativa, per exemple, restringint la
consulta de tècnics exclusivament als administradors.

El servei d'autenticació gestiona el registre i l'inici de sessió. Per al
registre, la contrasenya es protegeix amb bcrypt utilitzant 10 rondes de salt,
garantint que contrasenyes idèntiques produeixin hashos diferents. Per a
l'inici de sessió, es compara la contrasenya proporcionada amb el hash
emmagatzemat i, si coincideix, es genera un token JWT amb una expiració de
24 hores. En ambdós casos, la contrasenya mai es retorna al client, complint
amb els principis de protecció de dades del RGPD.

Un aspecte rellevant del disseny és l'ús de missatges d'error genèrics al
login: tant si l'email no existeix com si la contrasenya és incorrecta, el
missatge és sempre el mateix. Aquesta pràctica de seguretat estàndard evita
l'enumeració d'usuaris per part d'actors maliciosos.

7.1.3  Màquina d'estats amb XState

Per governar el cicle de vida de les incidències (RF-04), s'ha implementat
una màquina d'estats finits utilitzant la llibreria XState v5 amb l'API
setup(). Aquesta decisió respon a la necessitat de centralitzar i formalitzar
les regles de transició, evitant condicions disperses pel codi que serien
difícils de mantenir i verificar.

La màquina defineix cinc estats (Oberta, Assignada, En procés, Validada i
Tancada) i sis transicions, cadascuna protegida per guards que verifiquen
el rol de l'usuari que sol·licita el canvi. Per exemple, només un administrador
pot tancar definitivament una incidència, mentre que assignar-la pot fer-ho
qualsevol usuari que no sigui estudiant.

La integració amb la base de dades és especialment rellevant: quan un servei
necessita executar una transició, crea un actor XState, restaurant l'estat
actual des de la base de dades amb resolveState(). Abans d'executar la
transició, valida amb snapshot.can() si és permesa, permetent retornar
missatges d'error descriptius sense haver de modificar les dades.

Aquesta arquitectura permet un control d'accés dual: el middleware authorize()
protegeix rutes senceres, mentre que la màquina XState controla permisos
granulars per transició. Això garanteix que la lògica de negoci sobre qui
pot fer què estigui centralitzada en un únic punt, facilitant la seva
auditoria i modificació futura.

7.1.4  Endpoints implementats

En aquest Sprint s'han definit 8 endpoints agrupats en tres mòduls:
autenticació (registre i login, rutes públiques), incidències (creació, llistat,
detall i transició d'estat, rutes autenticades) i usuaris (perfil personal i
llistat de tècnics, amb RBAC addicional per als administradors).

Cada endpoint segueix un flux consistent: la ruta defineix el mètode HTTP i
la URL, els middlewares verifiquen l'autenticació i els permisos, el
controlador valida l'entrada i gestiona la resposta HTTP, i el servei executa
la lògica de negoci i les consultes a la base de dades.


--------------------------------------------------------------------------------
7.2  Sprint 3: Interfície Web d'Administració
--------------------------------------------------------------------------------

El tercer Sprint s'ha centrat en la construcció del frontend web, la interfície
que els administradors utilitzen per supervisar i gestionar el sistema
d'incidències. L'objectiu era proporcionar una Single Page Application (SPA)
funcional que permetés visualitzar estadístiques, gestionar incidències i
executar transicions d'estat directament des del navegador.

7.2.1  Configuració de l'entorn de desenvolupament

L'aplicació s'ha construït amb React 19, TypeScript i Tailwind CSS v4, servida
per Vite. La tria de Vite com a bundler respon a la seva rapidesa de compilació
i la integració nativa amb els plugins necessaris. S'ha configurat un proxy
que redirigeix les peticions /api al backend Express, eliminant completament
els problemes de CORS durant el desenvolupament i simplificant la configuració
de la comunicació entre ambdós servidors.

Tailwind CSS v4, a diferència de versions anteriors, no requereix fitxer de
configuració i detecta automàticament les classes utilitzades. Aquesta
integració directa amb Vite mitjançant el plugin @tailwindcss/vite ha eliminat
la necessitat de PostCSS i ha simplificat considerablement el pipeline de
construcció.

7.2.2  Capa de comunicació amb el backend

S'ha creat una instància centralitzada d'Axios amb dos interceptors que
automatitzen la gestió dels tokens JWT. L'interceptor de petició afegeix
automàticament el token d'autenticació al header Authorization de cada crida,
evitant duplicar aquesta lògica a cada servei. L'interceptor de resposta
detecta errors 401 (token expirat o invàlid), esborra el token local i
redirigeix a la pàgina de login, garantint una experiència d'usuari fluida
davant la caducitat de sessions.

Els serveis API s'organitzen per domini: autenticació (login i validació de
perfil), incidències (llistat, detall i transicions) i usuaris (llistat de
tècnics). Cada funció encapsula una crida HTTP concreta, tipada amb les
interfícies de TypeScript definides al mòdul de tipus.

7.2.3  Gestió de l'autenticació

L'estat d'autenticació es gestiona globalment mitjançant el patró React
Context. S'ha optat per Context en lloc de llibreries com Redux o Zustand
perquè l'únic estat global necessari és la sessió de l'usuari; afegir una
dependència addicional seria sobreenginyeria per a aquesta necessitat.

El context exposa quatre valors fonamentals: l'objecte d'usuari (o null si no
hi ha sessió), un booleà de càrrega, i les funcions de login i logout. Un
aspecte crític del login és que verifica que l'usuari tingui rol ADMIN abans
d'acceptar la sessió; els usuaris amb altres rols reben un missatge d'error
indicant que el panell és exclusiu per a administradors.

La protecció de rutes s'implementa mitjançant un component ProtectedRoute que
verifica l'estat d'autenticació i redirigeix a /login si l'usuari no és un
administrador autenticat. Aquesta és la primera capa de protecció (client-side);
la segona és el backend, que valida el JWT i el rol a cada petició.

7.2.4  Pàgines del panell d'administració

L'aplicació consta de cinc pàgines principals, organitzades dins d'un layout
comú amb una barra lateral de navegació fixa i una zona de contingut principal:

La pàgina de login presenta un formulari centrat amb el logotip de CityFix.
El dashboard mostra estadístiques generals del sistema amb visualitzacions
interactives. La llista d'incidències presenta una taula filtrable per estat,
sincronitzada amb la URL mitjançant useSearchParams per permetre compartir
enllaços filtrats directament. La pàgina de detall d'una incidència, la més
complexa del panell, es divideix en dues columnes: la principal amb la
descripció, imatges i comentaris, i la lateral amb informació estructurada
i botons d'acció segons l'estat actual.

El mapa de transicions vàlides (STATE_TRANSITIONS) es replica al frontend per
renderitzar correctament els botons d'acció sense consultar el backend. No
obstant, el backend continua validant cada transició amb XState, garantint
que la seguretat no depèn mai exclusivament del client.

7.3  Seguretat avançada: Registre per invitació i revocació d'accés
--------------------------------------------------------------------------------

Un cop establerta la infraestructura bàsica, s'ha dedicat una iteració
completa a tancar el model de seguretat del sistema. S'han identificat i
resolt dues vulnerabilitats crítiques del disseny inicial.

7.3.1  Registre per invitació (Invitation-based Registration)

El registre públic original acceptava el camp role directament del cos de la
petició, permetent que qualsevol persona s'autoassignés privilegis
d'administrador o tècnic. Això constituïa una vulnerabilitat de Mass
Assignment que calia corregir amb urgència.

S'ha implementat el patró de registre per invitació, inspirat en el model
Zero Trust: el registre d'estudiants és públic però el backend ignora el
camp role i força STUDENT programàticament. Per als rols privilegiats (ADMIN
i TECHNICAL), es requereix un token d'invitació generat prèviament per un
administrador. Aquests tokens es generen amb crypto.randomBytes(32), garantint
entropia criptogràfica real.

La creació de l'usuari privilegiat i el marcatge de la invitació com a
utilitzada s'executen dins d'una transacció atòmica de Prisma. Si una de les
dues operacions falla, cap de les dues s'aplica, prevenint inconsistències
com un token reutilitzable o un token perdut.

7.3.2  Revocació d'accés i protecció anti-deadlock

Per complementar el sistema d'invitacions, s'ha implementat un mecanisme de
revocació d'accés basat en soft delete: un camp active (booleà) al model
d'usuari que, quan es desactiva, impedeix el login sense trencar la
integritat referencial de les incidències i comentaris associats.

Per prevenir situacions irreversibles, s'han establert dues regles de
protecció:

La primera protegeix un administrador root (configurat via variable d'entorn)
que no pot ser revocat sota cap circumstància, garantint que sempre existeixi
almenys un punt d'accés al sistema.

La segona impedeix revocar l'últim administrador actiu mitjançant un comptatge
previ a la base de dades. Si només resta un administrador actiu, l'operació
es cancel·la, protegint el sistema contra una situació de deadlock
administratiu.

El camp used de la taula d'invitacions s'ha evolucionat d'un simple booleà a
un enum amb tres estats (Pendent, Utilitzada, Revocada), afegint traçabilitat
completa al cicle de vida de cada accés privilegiat.

7.4  Mòdul SIG: Visualització Geogràfica d'Incidències
--------------------------------------------------------------------------------

Un dels aspectes diferenciadors de CityFix és la integració d'un Sistema
d'Informació Geogràfica (SIG) que permet visualitzar les incidències sobre
un mapa interactiu del campus. Aquesta funcionalitat respon directament al
requisit RF-11 (Visualització cartogràfica) i a l'objectiu estratègic
d'anàlisi geolocalitzada de problemes.

7.4.1  Infraestructura geoespacial amb PostGIS

S'ha habilitat l'extensió PostGIS a Supabase per afegir capacitats
geoespacials a la base de dades PostgreSQL. PostGIS és l'estàndard de la
indústria per al tractament de dades geogràfiques en bases de dades
relacionals i s'ha escollit per la seva integració nativa amb Supabase, el
suport del tipus geography amb SRID 4326 (el mateix sistema de referència
que utilitzen GPS i OpenStreetMap) i la possibilitat de realitzar càlculs de
distància reals en metres si en el futur es necessiten consultes espacials
avançades.

Per desacoblar la interfície de l'emmagatzematge optimitzat, s'ha creat un
trigger PL/pgSQL que transforma automàticament els camps decimals latitude i
longitude en un punt binari PostGIS. L'aplicació treballa amb coordenades
decimals simples, mentre que la base de dades gestiona internament les
geometries optimitzades, seguint el principi de responsabilitat única.

7.4.2  Leaflet i les seves extensions

S'ha escollit Leaflet com a motor de mapes per tres raons fonamentals: és
completament lliure sota llicència BSD (eliminant costos i dependències de
serveis externs com Google Maps), és extraordinàriament lleuger (~42 KB
comprimit, menys d'una quarta part que OpenLayers) i disposa d'un ecosistema
madur de plugins.

S'han integrat dos plugins específics: leaflet.markercluster per a l'agrupació
visual de markers per proximitat amb animacions de dispersió, i leaflet.heat
per a la generació de mapes de calor amb gradient configurable. Les tiles
(imatges del mapa base) provenen d'OpenStreetMap, sense necessitat de clau
API ni configuració de facturació.

7.4.3  Format d'intercanvi GeoJSON

L'endpoint de dades geogràfiques retorna les incidències en format GeoJSON
(FeatureCollection), l'estàndard obert definit a l'RFC 7946. Aquesta decisió
garanteix la compatibilitat universal amb qualsevol eina GIS i facilita una
futura migració de la llibreria de mapes sense modificar l'API.

7.4.4  Clustering client-side

El pla inicial contemplava la possibilitat d'implementar clustering SQL al
servidor (ST_ClusterDBSCAN). Aquesta opció s'ha descartat intencionadament
en favor del clustering visual al frontend. La justificació rau en el volum
de dades esperat: en un campus universitari, el nombre d'incidències es mesura
en centenars, no en milers. Amb aquest volum, enviar tots els punts al client
i deixar que leaflet.markercluster els agrupi dinàmicament resulta més
eficient i ofereix una experiència d'usuari superior, amb animacions de
dispersió al fer zoom que el clustering precalculat no pot proporcionar.

7.4.5  Mapa de calor amb pes configurable

El mapa de calor no es limita a mostrar densitat: l'administrador pot
seleccionar el criteri de pes entre tres opcions (prioritat, densitat i
antiguitat) per obtenir perspectives analítiques diferents sobre les mateixes
dades. Per exemple, pot detectar que una zona no genera moltes incidències
però les poques que té són crítiques, o que una àrea acumula problemes
crònics sense resoldre.

El càlcul del pes es realitza al backend per garantir que el frontend no
hagi de conèixer la lògica de negoci i que les modificacions a la fórmula
de pes no requereixin canvis al client.

7.5  Mòdul d'Analítica i Visualització de Dades
--------------------------------------------------------------------------------

El darrer mòdul implementat fins al moment ha estat la transformació del
dashboard d'administració en una eina d'anàlisi de dades completa. S'ha
redissenyat completament la pàgina principal, passant de comptadors estàtics
a 9 visualitzacions interactives organitzades en 4 blocs temàtics.

7.5.1  Tria de Recharts

S'ha escollit Recharts com a llibreria de visualització per la seva integració
nativa amb React (cada gràfic és un component declaratiu), el suport dels
6 tipus de gràfic requerits sense plugins addicionals, la responsivitat
integrada i un pes de ~45 KB que la situa significativament per sota
d'alternatives com Highcharts o ECharts. S'ha descartat Chart.js pel seu
model imperatiu que trenca la filosofia declarativa de React, i D3.js
pur per ser excessivament complex per a aquest cas d'ús.

7.5.2  Blocs analítics

El Bloc Operatiu respon la pregunta «Quin és l'estat actual del sistema?»
mitjançant targetes KPI clicables i un gràfic de rosca que mostra la
distribució proporcional dels cinc estats. S'ha optat pel donut chart perquè
l'objectiu és percebre proporcions relatives, no comparar quantitats absolutes.

El Bloc Temporal analitza les tendències amb un gràfic d'àrees apilades que
mostra simultàniament l'evolució global i la composició per categoria, i un
gràfic de barres agrupades que compara les incidències creades amb les
tancades per setmana. Aquesta darrera visualització mesura la capacitat
d'absorció del sistema: si les barres de creació superen consistentment les
de tancament, el sistema està acumulant deute de manteniment.

El Bloc de Rendiment avalua l'eficiència dels recursos amb un gràfic de
barres horitzontals apilades per tècnic (que permet detectar colls d'ampolla),
un gràfic de dispersió que relaciona el temps de resolució amb la prioritat
(per auditar la qualitat del servei) i un gràfic de barres per categoria.

Finalment, el Bloc Social mostra un rànquing dels 10 usuaris que més
incidències han reportat, preparat per integrar-se amb el sistema de
gamificació previst al Sprint 7.

7.5.3  Endpoint unificat d'analítica

En lloc de crear 8 endpoints separats, s'ha optat per un únic endpoint que
executa totes les consultes en paral·lel amb Promise.all(). Aquesta decisió
redueix els roundtrips HTTP, garanteix la consistència temporal de les dades
(totes les mètriques es calculen al mateix instant) i simplifica la gestió
d'estat al frontend amb un sol hook d'efecte.

Les consultes d'agregació temporal requereixen la funció date_trunc() de
PostgreSQL, que Prisma no suporta nativament. Per a aquests casos específics
s'ha utilitzat SQL cru amb paràmetres posicionals per prevenir injeccions SQL.


--------------------------------------------------------------------------------
7.6  Tipificació de categories
--------------------------------------------------------------------------------

Durant el desenvolupament s'ha detectat que el camp de categoria de les
incidències, inicialment de text lliure, generava inconsistències en les
agregacions i filtres. S'ha substituït per un enum Prisma amb 9 valors
predefinits (Il·luminació, Mobiliari urbà, Via pública, Neteja, Zones verdes,
Senyalització, Accessibilitat, Tecnologia i Altres) que cobreixen la
tipologia d'infraestructures d'un campus universitari. Al frontend, un mapa
de traducció converteix cada valor de l'enum a la seva etiqueta en català.


================================================================================
8  ASPECTES TÈCNICS
================================================================================

8.1  Estructura de directoris del projecte

Per a la correcta compartimentació dels arxius, el projecte s'organitza en
dos directoris principals que reflecteixen la separació client-servidor:

El backend segueix el patró Routes-Middleware-Controller-Service amb les
carpetes config/ (connexió a base de dades i variables d'entorn), controllers/
(validació d'entrada i resposta HTTP), machines/ (màquina d'estats XState),
middlewares/ (autenticació JWT i RBAC), routes/ (definició d'endpoints),
services/ (lògica de negoci i consultes Prisma) i types/ (interfícies
TypeScript compartides).

El frontend segueix una estructura orientada a funcionalitat amb api/ (serveis
HTTP amb Axios), components/ (elements reutilitzables com badges d'estat i
prioritat, components del mapa), context/ (gestió d'autenticació global),
hooks/ (hooks personalitzats), pages/ (pàgines de l'aplicació) i types/
(interfícies mirall del backend).

8.2  Estructura de la base de dades

La base de dades és PostgreSQL allotjada a Supabase, amb l'extensió PostGIS
per a les capacitats geoespacials. L'esquema es defineix amb Prisma ORM, que
proporciona tipatge automàtic, migracions versionades i un client generat.

Les taules principals són:

- users: Emmagatzema els usuaris del sistema amb rol (STUDENT, TECHNICAL,
  ADMIN), punts de gamificació i estat actiu. Inclou el vincle optional amb
  la invitació que va originar l'accés per als rols privilegiats.

- reports: Les incidències reportades, amb estat governat per la màquina
  XState, prioritat, categoria tipificada, coordenades GPS, vincles amb
  el creador i el tècnic assignat, i marca temporal de resolució.

- invites: Les invitacions per a registre privilegiat, amb token criptogràfic,
  estat (Pendent/Utilitzada/Revocada) i relació amb l'usuari creat.

Les relacions principals definides són: un usuari pot crear múltiples
incidències; un tècnic pot tenir múltiples incidències assignades; una
incidència pot tenir múltiples imatges i comentaris; i cada usuari privilegiat
està vinculat opcionalment a una invitació.

8.3  Tecnologies del projecte

Backend:
- Node.js amb Express v5: servidor HTTP i routing.
- TypeScript: tipatge estàtic per a la detecció d'errors en compilació.
- Prisma v7: ORM amb driver adapter PrismaPg per a PostgreSQL.
- XState v5: màquina d'estats finits per al cicle de vida d'incidències.
- jsonwebtoken i bcrypt: autenticació JWT i hashing de contrasenyes.
- PostgreSQL amb PostGIS: base de dades relacional amb extensió geoespacial.
- Supabase: plataforma de backend-as-a-service per a l'allotjament.

Frontend:
- React 19: llibreria UI declarativa basada en components.
- Vite: bundler amb compilació instantània i Hot Module Replacement.
- Tailwind CSS v4: framework CSS utility-first sense configuració.
- Axios: client HTTP amb interceptors per a JWT automàtic.
- React Router v7: routing client-side amb guards de ruta.
- Recharts: visualització de dades basada en D3.js.
- Leaflet + react-leaflet: mapes interactius amb clustering i heatmaps.

8.4  Decisions tècniques rellevants

- RBAC dual: el control d'accés opera a dos nivells complementaris. El
  middleware authorize() protegeix rutes senceres, mentre que la màquina
  XState controla permisos granulars per transició.

- Soft delete per a la revocació: en lloc d'eliminar els usuaris revocats,
  un camp booleà active permet desactivar-los preservant la integritat
  referencial de totes les dades associades.

- Transacció atòmica per al registre privilegiat: la creació de l'usuari i
  el marcatge de la invitació s'executen com una unitat indivisible.

- Trigger PostGIS: desacobla la interfície (coordenades decimals) de
  l'emmagatzematge optimitzat (geometria PostGIS) sense que l'aplicació
  necessiti conèixer PostGIS.

- Endpoint unificat d'analítica: redueix roundtrips HTTP i garanteix
  consistència temporal entre les mètriques del dashboard.

- Clustering client-side: adequat al volum de dades d'un campus i superior
  en interactivitat al clustering precalculat al servidor.

9  PROVES I VALIDACIÓ

9.1  Validació d'endpoints

La validació dels endpoints de l'API REST s'ha dut a terme mitjançant Postman,
una eina que permet enviar peticions HTTP i analitzar les respostes del
servidor. S'ha verificat manualment cada endpoint definit, comprovant que les
dades retornades i les accions realitzades fossin les esperades.

Per a cada endpoint, s'ha simulat el flux complet: registre d'un usuari, inici
de sessió per obtenir el token JWT, i execució de les operacions protegides
amb el token al header Authorization. S'han verificat específicament els
casos límit del control d'accés: un estudiant intentant executar transicions
reservades a administradors, un tècnic intentant accedir a rutes d'admin, i
la correcta validació de la màquina d'estats davant transicions no permeses.

9.2  Validació de la màquina d'estats

La integració de XState amb la base de dades s'ha validat de forma exhaustiva,
recorrent tots els camins possibles de la màquina d'estats: des de la creació
d'una incidència (OPEN) fins al seu tancament (CLOSED), passant per les
reassignacions (ASSIGNED → OPEN) i els rebutjos (VALIDATED → IN_PROGRESS).
S'ha verificat que els guards de rol funcionen correctament per a tots els
perfils d'usuari i que els missatges d'error són descriptius i específics.

9.3  Validació del frontend

La interfície web s'ha validat mitjançant proves manuals en navegadors Chrome
i Firefox, comprovant la correcta renderització de tots els components, la
funcionalitat dels filtres, les transicions d'estat des del detall de les
incidències, i la protecció de rutes per als usuaris no autoritzats.

S'ha verificat especialment el flux de sessió: caducitat del token amb
redirecció automàtica al login, revalidació del token en recarregar la pàgina,
i el correcte funcionament del doble nivell de verificació de rol ADMIN.

10  RESULTATS

El desenvolupament realitzat fins al moment ha produït una plataforma funcional
que cobreix els requisits fonamentals del sistema:

10.1  API REST completa

S'ha implementat una API amb més de 15 endpoints que cobreixen l'autenticació
amb JWT, la gestió del cicle de vida d'incidències amb màquina d'estats, el

10.2  Panell d'administració web

El frontend web permet als administradors supervisar l'estat del sistema amb
un dashboard analític de 9 visualitzacions, gestionar el cicle de vida complet
de les incidències amb transicions d'estat directes, visualitzar la distribució
geogràfica amb mapes interactius de markers i calor, administrar els accessos
privilegiats amb invitacions i revocacions, i filtrar i analitzar les dades
des de múltiples perspectives temporals i temàtiques.

10.3  Model de seguretat robust

El sistema implementa cinc capes de seguretat complementàries: autenticació
JWT amb caducitat, RBAC per middleware i per màquina d'estats, registre per
invitació criptogràfica, revocació d'accés amb soft delete, i protecció
anti-deadlock per a l'administrador root.
